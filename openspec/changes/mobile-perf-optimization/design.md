## Context

The application is a mobile-first vocabulary learning SPA. It runs entirely in the client's browser (local-first via IndexedDB). To ensure the application is lightweight, quick responsive, and highly tactile, we must optimize styling and assets rendering pipelines. Mobile devices are extremely sensitive to GPU-heavy filters (like `backdrop-filter`) and excessive rendering calculations (like `transition: all`). Additionally, to support offline study conditions, a zero-dependency service worker is needed.

## Goals / Non-Goals

**Goals:**
- **Smooth 60fps/120fps Scrolling & Animations**: Remove all render-blocking blur filters and generic CSS transitions that cause rendering lag on low-end mobile devices.
- **Zero-Dependency Hand-Crafted Service Worker**: Implement offline assets caching natively in a single custom Service Worker file (`sw.js`), achieving 100% offline functionality with zero external dependencies.
- **Fluid Gesture-Driven Study**: Implement lightweight, hardware-accelerated touch swipe gestures (left/right) on flashcards using native React touch event handlers.
- **Bandwidth Reduction**: Optimize index.html web fonts preload size.

**Non-Goals:**
- Using third-party frameworks like Workbox or offline-plugin which would bloat the bundle size.
- Incorporating complex swipe motion libraries (e.g. Framer Motion, React Spring).
- Adding complex gesture-based features in screens other than the flashcard Learn view.

## Decisions

### 1. Flat CSS Cards over Backdrop Blur
- **Decision**: Replace `backdrop-filter: blur(16px)` and `-webkit-backdrop-filter` in CSS with solid opaque colors (`var(--bg-card-solid)`) and thin borders (`1px solid var(--border-color)`).
- **Rationale**: Dynamic CSS filters are extremely expensive for mobile GPUs as they trigger offscreen render-pass layers on every scrolling/painting frame. Flat cards with clean borders require zero offscreen passes and keep frame rates high.
- **Alternatives Considered**: Reducing the blur radius. *Rejected* because even small blur values degrade mobile GPU performance significantly during heavy scroll gestures.

### 2. Transition Target Specificity
- **Decision**: Replace `transition: all 0.3s ease` and similar open-ended transition rules with explicit property listings (e.g. `transition: transform 0.3s cubic-bezier(0.4, 0, 0.2, 1), background-color 0.2s ease, border-color 0.2s ease`).
- **Rationale**: When using `transition: all`, the browser's layout engine has to watch and recalculate properties for all style changes on every frame, which triggers rendering pipeline thrashing. Specific property targets skip this check.

### 3. Font weight Pruning
- **Decision**: Prune the Outfit Google Fonts load from 5 weights down to 3 weights: 400 (Regular), 600 (Semi-bold), and 700 (Heavy).
- **Rationale**: Each extra weight represents another WOFF2 file that blocks layout/rendering on slow cellular networks. Three weights are fully sufficient for clean typography and reduce font package size by ~40%.

### 4. Custom Hand-Written Service Worker (`sw.js`)
- **Decision**: Code a direct native service worker inside the `/public` folder with a standard `Cache-First` strategy for static resources (CSS, JS, SVG, Fonts) and `Network-First` strategy for `index.html`.
- **Rationale**: Hand-crafting a raw Service Worker has zero bundle overhead (0kB external libraries) and compiles instantly, bypassing UNC file copy issues or Webpack/Vite plugin complications.
- **Alternatives Considered**: Using `@vite-pwa/plugin` or `Workbox`. *Rejected* due to complexity, package installation size, and potential UNC socket bugs on Vite preview maps.

### 5. Pure React GPU-composited Gesture Handling
- **Decision**: Implement native `onTouchStart`, `onTouchMove`, and `onTouchEnd` handlers on the flashcard container. Compute swipe displacements and map them directly to `transform: translate3d(deltaX, 0, 0) rotate(deltaX * 0.05deg)` inline CSS styles.
- **Rationale**: Direct CSS `translate3d` and `rotate` transforms run exclusively on the GPU compositor layer without triggering browser repaint or reflow, ensuring maximum scrolling fluidness on any viewport.
- **Alternatives Considered**: Bulky drag libraries (e.g., React UseGesture). *Rejected* because they introduce multiple dependencies and increase the JS package size.

## Risks / Trade-offs

- **[Risk] Service Worker Cache Staling**: Users might not receive new updates of CSS/JS if cached too aggressively.
  - *Mitigation*: We will use hashed asset names generated by Vite builds (which change on every build) and cache-bust by listening to the service worker's `install` and `activate` events to instantly delete legacy cache tables on new version releases.
- **[Risk] Gesture Jitter**: Touch tracking might conflict with native mobile page scrolling.
  - *Mitigation*: Prevent default behavior only on the vertical/horizontal axis of the active card when swiping is in progress, and use standard threshold boundaries (e.g., minimum 50px movement) to identify valid swipes.
